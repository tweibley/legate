# Plan: Add Markdown Documentation to Web UI

This plan outlines the steps to integrate a documentation section into the ADK Web UI, allowing Markdown files stored in `public/docs` to be rendered and displayed.

## 1. Dependency Setup

*   **Add Kramdown Gem**:
    *   Open `Gemfile`.
    *   Add the line: `gem 'kramdown'`
    *   Run `bundle install` in the terminal to install the gem.

## 2. Create Documentation Directory

*   Create a new directory: `public/docs`. This is where the Markdown documentation files will reside.
    *   Initially, we can add a placeholder file like `public/docs/introduction.md` with some basic markdown content for testing.

## 3. Application Logic (`lib/adk/web/app.rb`)

*   **Require Kramdown**:
    *   Add `require 'kramdown'` at the top of `lib/adk/web/app.rb` with other requires.
*   **Helper Method for Markdown Rendering**:
    *   Create a helper method, e.g., `render_markdown(file_path)`, that:
        *   Takes a file path to a Markdown document.
        *   Reads the file content.
        *   Uses `Kramdown::Document.new(markdown_content).to_html` to convert it to HTML.
        *   Returns the HTML string.
        *   Handles potential errors (e.g., file not found).
*   **Helper Method for Document Summary (Optional but Recommended)**:
    *   Create a helper method, e.g., `generate_summary(markdown_content, max_lines = 5)`, that:
        *   Takes markdown content as a string.
        *   Extracts the first few lines (e.g., up to `max_lines` or until a blank line after some content).
        *   Potentially strips markdown formatting for a plain text summary.
        *   Returns a short summary string.
*   **Route for Documentation Index (`GET /docs`)**:
    *   This route will:
        *   List all `.md` files in the `public/docs` directory.
        *   For each file, generate a title (e.g., from the filename, or potentially from the first H1 tag if we want to get fancy later).
        *   Generate a short summary using the `generate_summary` helper.
        *   Pass a list of document objects (e.g., `[{title: "Introduction", filename: "introduction.md", summary: "..."}, ...]`) to a new Slim view.
        *   Render `docs_index.slim`.
*   **Route for Displaying a Single Document (`GET /docs/:filename`)**:
    *   This route will:
        *   Take `filename` from the path. Ensure it's a valid filename and doesn't allow directory traversal (e.g., sanitize input).
        *   Construct the full path to `public/docs/#{filename}.md`.
        *   Check if the file exists. If not, show a 404 error.
        *   Use the `render_markdown` helper to convert its content to HTML.
        *   Pass the HTML content and the document title to a new Slim view.
        *   Render `docs_show.slim`.

## 4. Create Views (`lib/adk/web/views/`)

*   **`docs_index.slim`**:
    *   This view will:
        *   Display a title like "Documentation".
        *   Iterate through the list of document objects passed from the `/docs` route.
        *   For each document, display its title as a link to `/docs/:filename` (without the `.md` extension in the link, the route can add it back).
        *   Display the short summary for each document.
        *   Use Bulma styling for a clean presentation (e.g., cards or a list).
*   **`docs_show.slim`**:
    *   This view will:
        *   Display the title of the document.
        *   Render the HTML content (passed from the `/docs/:filename` route) using `==` in Slim to output raw HTML.
        *   Ensure basic styling for the rendered HTML content. We might need to add some CSS rules for common Markdown elements (headers, paragraphs, lists, code blocks) if Bulma doesn't cover them sufficiently out-of-the-box from Kramdown's output.
*   **`error_404.slim` (Enhancement)**:
    *   Ensure the existing `error_404.slim` or a new one can be used to display a "Document not found" message.

## 5. Update Main Layout

*   **Add Navigation Link**:
    *   In `lib/adk/web/views/layout.slim`, add a new navigation link in the sidebar or navbar (e.g., "Help" or "Documentation") that points to `/docs`.

## 6. Styling (CSS)

*   Review the appearance of the rendered Markdown in `docs_show.slim`.
*   If necessary, add custom CSS rules to `public/css/application.css` (or a new SASS file that compiles to it) to style HTML elements generated by Kramdown (e.g., `h1`, `h2`, `p`, `ul`, `ol`, `pre`, `code`, `blockquote`) to ensure they look good within the existing UI theme.

## 7. Testing

*   Create a few sample Markdown files in `public/docs`.
*   Navigate to `/docs` to see the index.
*   Click on links to view individual documents.
*   Test edge cases (e.g., non-existent document).
*   Check responsiveness and styling.
