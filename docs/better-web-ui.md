# ADK Ruby Web UI Enhancement Plan

This document outlines planned improvements for the ADK Ruby Web UI (`lib/adk/web/app.rb` and associated views/partials) to make it more useful for developers using and building upon the ADK library.

## Enhancement Checklist

### 1. Debugging & Introspection

-   [x] **Show Agent Plan/Tool Calls in Chat:**
    -   **Goal:** Allow developers to see the step-by-step tool execution plan decided by the agent during a chat conversation.
    -   **Implementation:**
        -   Modify `ADK::Agent#run_task` (or its internal methods) to capture the plan (sequence of tool calls, parameters, intermediate results). This might involve adding more detailed logging within the planning/execution loop or storing the plan temporarily.
        -   Update the `ADK::Session` or `ADK::Event` structure to potentially hold this plan information alongside the final agent response.
        -   Modify the `POST /agents/:name/chat` route in `app.rb` to retrieve this plan information after `agent.run_task` completes.
        -   Update the `_chat_message.slim` partial (and potentially the `process_agent_response` helper) to render the plan details. This could be hidden by default and revealed by a "Show Details" toggle/button next to the agent's message in the chat UI.
        -   Consider how to handle multi-turn plans or complex interactions.

-   [ ] **View Agent-Specific Logs:**
    -   **Goal:** Provide easy access to recent log entries specifically related to a running agent instance.
    -   **Implementation:**
        -   Enhance the ADK logging setup to allow tagging log entries with the relevant agent name.
        -   Create a new route (e.g., `GET /agents/:name/logs`) that reads the log file (or potentially a dedicated logging sink) and filters entries for the specified agent.
        -   Display these logs on the agent detail page (`/agents/:name`, `views/agent.slim`) perhaps in a collapsible section or via an HTMX request triggered by a button.
        -   Consider limiting the number of log lines shown initially and providing pagination or filtering options.

-   [ ] **MCP Test Utility:**
    -   **Goal:** Allow testing MCP server connections and tool listings independently of agent configuration.
    -   **Implementation:**
        -   Create a new route (e.g., `GET /mcp_test`) and a corresponding view (`views/mcp_test.slim`).
        -   The view should provide form fields for entering MCP connection details (type, command/URL, name).
        -   Add a `POST /mcp_test/fetch` route that takes the form data, uses the existing `fetch_mcp_tools` helper in `app.rb`, and returns an HTML fragment displaying the results (success/error, tool list) using HTMX.
        -   Link to this utility from relevant places like the main `/tools` page or the agent creation/edit forms.

### 2. Agent Definition Management

-   [ ] **Import/Export Agent Definitions:**
    -   **Goal:** Enable saving and loading agent configurations to/from files.
    -   **Implementation:**
        -   **Export:** Add an "Export All" button on `/agents` and an "Export" button on `/agents/:name`. Create routes (e.g., `GET /agents/export`, `GET /agents/:name/export`) that fetch definition(s) from the `DefinitionStore`, format them as JSON (or YAML), set appropriate `Content-Type` and `Content-Disposition` headers, and return the file content.
        -   **Import:** Add an "Import Agents" button/form on `/agents`. Create a `POST /agents/import` route that accepts a file upload. Parse the file (JSON/YAML), validate the structure for each definition, and use `DefinitionStore#save_definition` to save them. Provide feedback on success/failure via HTML response/HTMX.

-   [ ] **Clone Agent:**
    -   **Goal:** Quickly create a new agent definition based on an existing one.
    -   **Implementation:**
        -   Add a "Clone" button to each row in the `/agents` table (`_agent_row.slim`).
        -   This button could trigger a GET request to a new route like `GET /agents/:name/clone_form` which pre-populates the standard new agent form (`_new_agent_form.slim`) with the details of the agent being cloned (perhaps appending "_copy" to the name).
        -   Alternatively, the button could directly trigger a `POST /agents/:name/clone` route which fetches the definition, modifies the name, saves the new definition via the `DefinitionStore`, and returns the new agent row via HTMX OOB swap, similar to the standard create route.

### 3. Task Execution & Development

-   [ ] **Load Example Task into Executor:**
    -   **Goal:** Make it trivial to test the examples generated by the "Generate Example Task" feature.
    -   **Implementation:**
        -   Modify the `GET /agents/:name/generate_example_task` route: Instead of just returning JSON, return an HTML fragment containing the pretty-printed JSON *and* a "Load into Executor" button.
        -   Use Javascript (triggered by the new button) to take the JSON content from the fragment and place it into the `task_json` textarea on the agent detail page (`views/agent.slim`).

-   [ ] **Consistent Tool Parameter Display:**
    -   **Goal:** Ensure tool parameters (type, description, required/optional status) are clearly and consistently displayed across all relevant views.
    -   **Implementation:**
        -   Review and update the parameter rendering logic in:
            -   `views/tools.slim` (main tools page)
            -   `views/agent.slim` (agent detail tool list)
            -   `views/_agent_tool_table.slim` (tool table partial used in agent detail)
            -   `views/_edit_agent_tools.slim` (tool selection during edit)
        -   Ensure the data passed to these views (e.g., `@native_tools`, `view_configured_tools`, `@all_available_tools`) consistently includes the `required` flag for each parameter. (Note: recent code changes seem to have addressed parameter structure conversion, but double-check rendering).

### 4. UI/UX

-   [ ] **Consistent Toast Notifications:**
    -   **Goal:** Provide clear, non-intrusive feedback for background actions.
    -   **Implementation:**
        -   Identify key actions: Agent Create/Delete/Update, Start/Stop, Task Execution Success/Error (especially for background/MCP errors not handled by direct HTML swaps).
        -   Define standard Javascript functions (or leverage an existing library if one is added) triggered by HTMX `HX-Trigger` headers (e.g., `showSuccessToast`, `showErrorToast`).
        -   Modify the relevant backend routes (`POST /agents`, `DELETE /agents/:name`, `PUT /agents/:name/update/:field`, `POST /agents/:name/start`, `POST /agents/:name/stop`, etc.) to include the appropriate `HX-Trigger` header in their responses upon success or specific error conditions.

-   [ ] **Search/Filter Agents:**
    -   **Goal:** Make it easier to find specific agents when the list grows large.
    -   **Implementation:**
        -   Add a search input field above the agents table on `/agents` (`views/agents.slim`).
        -   Use Javascript to filter the table rows client-side as the user types OR implement a server-side filter:
            -   **Client-side:** Simple, good for moderate numbers of agents.
            -   **Server-side:** Add an HTMX attribute (`hx-get`, `hx-trigger="keyup changed delay:500ms"`, `hx-target="#agent-table-body"`) to the search input that sends the query to a route like `GET /agents/search?query=...`. This route would fetch filtered agents from the `DefinitionStore` and return only the `<tbody>` content for the table.
