# ADK Ruby Application Deployment Checklist (GCP Cloud Run)

This checklist provides a high-level overview of the steps involved in deploying your ADK Ruby application to Google Cloud Platform (GCP) using Cloud Run, leveraging the assets generated by the `adk generate deployment` command.

Refer to the detailed guide (`deployment/README-GCP-DEPLOYMENT.md` after generation, or `docs/go-to-gcp-production-gemini.md` in the source) for comprehensive explanations and commands.

## ☐ Phase 1: Prerequisites

*   [ ] **GCP Account & Project:** Ensure you have an active Google Cloud Platform account with a project created and billing enabled.
*   [ ] **`gcloud` CLI:** Install and authenticate the Google Cloud SDK (`gcloud auth login`, `gcloud config set project YOUR_PROJECT_ID`).
*   [ ] **Docker:** Install Docker Desktop or Docker Engine locally.
*   [ ] **ADK Application:** Have your ADK Ruby application code ready.
*   [ ] **API Keys:** Obtain necessary API keys (e.g., for your language model provider).

## ☐ Phase 2: Generate Deployment Assets

*   [ ] **Run Generator Command:** Execute the ADK CLI command in your project's root directory:
    ```bash
    adk generate deployment \\
        --cloud gcp \\
        --entry-point path/to/your/main_script.rb \\
        --agent-entry-points path/to/agent1.rb,path/to/agent2.rb \\ # Optional
        --name my-adk-deployment \\ # Optional: Output directory name
        --generate-sample-entrypoint \\ # Optional: Creates bin/adk_web_entrypoint.rb
        --gcp-project-id YOUR_PROJECT_ID \\
        --gcp-region YOUR_GCP_REGION # Optional: defaults to us-central1
        # Add other --gcp-* options as needed
    ```
*   [ ] **Locate Assets:** A directory (e.g., `my-adk-deployment/`) should be created containing:
    *   `Dockerfile` (for main entry point)
    *   `Dockerfile.agent.*` (if `--agent-entry-points` was used)
    *   `.dockerignore`
    *   `deploy-gcp.sh` (deployment script)
    *   `README-GCP-DEPLOYMENT.md` (detailed guide)
    *   Potentially `bin/adk_web_entrypoint.rb` (if `--generate-sample-entrypoint` was used)

## ☐ Phase 3: Review and Customize

*   [ ] **Review `Dockerfile`(s):** Check the base image and system dependencies. Add any extra `apt-get` packages your application might need.
*   [ ] **Review `.dockerignore`:** Ensure all necessary files/directories are excluded from the Docker build context.
*   [ ] **Review `deploy-gcp.sh`:**
    *   Carefully examine the **Configuration** section at the top.
    *   Verify `PROJECT_ID`, `REGION`, resource names (`REDIS_INSTANCE_NAME`, `MAIN_SERVICE_NAME`, `AR_REPO_NAME`, `CONNECTOR_NAME`, `SECRET_NAME`). Ensure `SECRET_NAME` is set to `google-api-key` or your preferred name.
    *   Adjust memory/CPU (`MAIN_MEMORY`, `MAIN_CPU`) if needed.
    *   **Important:** Decide on and configure the `RUN_SERVICE_ACCOUNT`. Using a dedicated service account is **strongly recommended** for security. **Action:** Create a dedicated Service Account in GCP *before* running the script, grant it necessary IAM roles (e.g., `roles/secretmanager.secretAccessor`, `roles/redis.client`), and update the `RUN_SERVICE_ACCOUNT` variable in the script with its email address. If left empty, the less secure default Compute Engine SA will be used (ensure *it* has the needed permissions).
    *   Review the sections for building/deploying agent services (if applicable) and uncomment/modify them as needed.
*   [ ] **Consider Health Checks:** Determine if your main service needs HTTP health checks (startup/liveness probes). If so, implement a suitable endpoint (e.g., `/healthz`) in your application code. **Tip:** If you used `--generate-sample-entrypoint`, the generated `bin/adk_web_entrypoint.rb` provides this endpoint. Modify `deploy-gcp.sh` to include `--probe-liveness` / `--probe-startup` flags if needed.

## ☐ Phase 4: Execute Deployment

*   [ ] **Navigate to Directory:** `cd my-adk-deployment/` (or your chosen output directory name).
*   [ ] **Run Deployment Script:** Execute the generated script:
    ```bash
    ./deploy-gcp.sh
    ```
*   [ ] **Monitor Output:** Watch the script output for progress and potential errors.
*   [ ] **Provide Secret:** If the API key secret (default name `google-api-key`) doesn't exist, the script will prompt you to enter the value securely.
*   [ ] **Wait for Completion:** The script handles API enablement, resource creation (Redis, VPC Connector, Artifact Registry repo), image building/pushing, and Cloud Run deployment. This may take several minutes.

## ☐ Phase 5: Verify Deployment

*   [ ] **Check Service URL:** Note the Cloud Run service URL output by the script. Access it in your browser (if it's the web UI and `--allow-unauthenticated` was used).
*   [ ] **Check GCP Console:**
    *   Verify the Cloud Run service(s) are running and healthy.
    *   Confirm the Memorystore Redis instance is active.
    *   Check that the VPC Access Connector is active.
    *   Check Health Check status for the service(s) if configured.
    *   Look at Artifact Registry to see your pushed container image(s).
    *   Verify the secret (`google-api-key` or your chosen name) was created in Secret Manager.
*   [ ] **Check Logs:** Use `gcloud logging read ...` (see Troubleshooting section in the detailed guide) or the GCP Console Logs Explorer to check for any application errors in your Cloud Run service(s).
*   [ ] **Test Functionality:** Interact with your deployed ADK application to ensure it's working as expected and can connect to Redis and use the API key (`GOOGLE_API_KEY` env var).

---

*This checklist is a guide. Your specific application might require additional steps or configurations.*
